# 管理端退款与订单操作矩阵

本文是模拟业务规则，用于 RAG 检索和 Agent 评测，不是线上财务政策。最终退款资格、金额和支付渠道必须由 Java 业务服务计算。

| 场景 | 建议动作 | Agent 可做 | 是否确认 | 备注 |
| --- | --- | --- | --- | --- |
| 门店未接单前取消 | 查询订单并提出取消 | 查询、生成取消预览 | 必须 | 退款金额以业务服务为准 |
| 门店已接单但未出餐 | 联系门店确认 | 查询、转人工 | 必须 | 可能产生取消费用 |
| 配送异常未送达 | 查询配送轨迹 | 查询、生成售后单预览 | 必须 | 不得直接承诺全额退款 |
| 少件/错件 | 记录问题并核验订单明细 | 查询、生成补发/退款预览 | 必须 | 需要订单和图片等证据 |
| 重复扣款 | 查询支付流水摘要 | 查询、转财务 | 必须 | 不暴露完整支付信息 |
| 批量退款 | 查询符合条件的订单 | 生成批量预览 | 二次审批 | 默认禁止 Agent 直接执行 |

任何退款、补发、订单状态修改都必须携带确认 token、幂等键和审计上下文，并由 Java 端进行最终权限和业务规则校验。
